Derivacin de caractersticas

  1. Qu ventajas tiene el anlisis de una URL contra el anlisis de otros datos, cmo el tiempo de vida del dominio, o las caractersticas de la pgina Web?
  • Extraccin: Los features de una URL son fciles de extraer y analizar, en comparacin con otras mtricas, comom lo es el tiempo de vida del dominio.
  • Accesibilidad: Las URLs son pblicas y accesibles, lo que nos permite un anlisis sin requerir acceso a informacin privada.
  • Rapidez: La URL proporciona informacin directa sobre una direccin web, facilitando la evaluacin inmediata de su validez.
  1. Qu caractersticas de una URL son ms prometedoras para la deteccin de phishing?
  • Shorted URL: Los atacantes usan URLs acortadas para ocultar el destino real del enlace, logrando que los usuarios tengan ms dificultal al identificar sitios maliciosos.
  • HTTPS: Los sitios de phishing a menudo no utilizan HTTPS, o no tienen los certificados implementados de manera correcta.
  • Direcciones IP puesta en el nombre del dominio: Los atacantes a menudo usan direcciones IP (http://192.168.1.1/...) en lugar de nombres de dominio, ya que dificulta la asociacin con una entidad legtima.
  • Uso de caracteres especiales y patrones aleatorios:La presencia de caracteres inusuales, guiones, nmeros o combinaciones de letras que aumentan la entropa del nombre de dominio o de la ruta puede indicar que la URL fue generada automticamente.
  • Coincidencia de marca: La similitud del dominio con nombres de marcas reconocidas (por ejemplo, micr0soft en lugar de microsoft) o el uso de caracteres visualmente similares. Los atacantes a menudo manipulan letras para que la URL parezca legtima.

def has_random_numeric_strings(url):
    pattern = re.compile(r'\d{5,}')  # sequences of 5 or more digits
    matches = re.findall(pattern, url)
    return len(matches)

def uses_https(url):
    parsed_url = urlparse(url)
    return 1 if parsed_url.scheme == 'https' else 0

def number_letter_ratio_in_path(url):
    parsed_url = urlparse(url)
    path = parsed_url.path
    letters = sum(c.isalpha() for c in path)
    numbers = sum(c.isdigit() for c in path)
    if numbers == 0: 
        return 0
    return letters / numbers

def letter_ratio_in_domain(url):
    parsed_url = urlparse(url)
    domain = parsed_url.netloc
    letters = sum(c.isalpha() for c in domain)
    if len(domain) == 0: 
        return 0
    return letters / len(domain)


def sensitive_keywords(url):
    sensitive_words = ["login", "password", "bank", "paypal", "account", "secure", "verify"]
    count = sum(url.lower().count(word) for word in sensitive_words)
    return count

def domain_age(url): 
    try:
        parsed_url = urlparse(url)
        domain = parsed_url.netloc
        domain_info = whois.whois(domain)

        creation_date = domain_info.creation_date

        if isinstance(creation_date, list):
            creation_date = creation_date[0]
        age = (datetime.now() - creation_date).days

        return int(age)

    except Exception as e:
        return 0

def count_hyphens_and_underscores(url):
    hyphen_count = url.count("-")
    underscore_count = url.count("_")
    return hyphen_count + underscore_count

def check_suspicious_tld(url):
    suspicious_tlds = [".tk", ".ml", ".ga", ".cf", ".gq"]  # suspicious TLDs
    parsed_url = urlparse(url)
    domain = parsed_url.netloc
    tld = domain.split(".")[-1]

    if "." + tld in suspicious_tlds:
        return 1  
    else:
        return 0  

def url_length(url):
    return len(url)

def subdomain_count(url):
    parsed_url = urlparse(url)
    return len(parsed_url.netloc.split('.')) - 2 

def vowel_consonant_ratio (x):
    x = x.lower()
    vowels_pattern = re.compile('([aeiou])')
    consonants_pattern = re.compile('([b-df-hj-np-tv-z])')
    vowels = re.findall(vowels_pattern, x)
    consonants = re.findall(consonants_pattern, x)
    try:
        ratio = len(vowels) / len(consonants)
    except: 
        ratio = 0
    return ratio

def count_redirects(url):
    response = requests.get(url, allow_redirects=False)
    return len(response.history)

def extract_path_extensions(url):
    parsed_url = urlparse(url)
    path = parsed_url.path
    return len([part.split('.')[-1] for part in path.split('/') if '.' in part])

def count_special_characters(url):
    special_character_pattern = re.compile(r'[^a-zA-Z0-9]')
    special_characters = re.findall(special_character_pattern, url)
    return len(special_characters)

def domain_length(url):
    parsed_url = urlparse(url)
    domain = parsed_url.netloc
    return len(domain)

Discusin

  • Cul es el impacto de clasificar un sitio legtimo como phishing? Un falso positivo puede tener muchos impactos negativos:

    • Afecta la experiencia del usuario: Los usuarios legtimos pueden ser bloqueados o recibir advertencias innecesarias al intentar acceder a sitios confiables.
    • Prdida de productividad: Si se clasifican incorrectamente sitios internos de la empresa como phishing, los empleados pueden perder acceso a recursos importantes.
    • Prdida de credibilidad del sistema: Un modelo con muchos falsos positivos podra generar desconfianza en el equipo de seguridad, haciendo que los usuarios lo ignoren.

  • Cul es el impacto de clasificar un sitio de phishing como legtimo?

    • Aumento del riesgo de ataques exitosos: Los usuarios podran ingresar credenciales o descargar malware creyendo que el sitio es seguro.
    • Prdida de datos y brechas de seguridad: Los atacantes pueden robar informacin confidencial de la empresa o de los usuarios.
    • Impacto financiero: Un ataque exitoso puede generar prdidas econmicas significativas, desde fraudes hasta costos de recuperacin de sistemas.
    • Prdida de confianza: Si el modelo no detecta ataques de phishing, los empleados podran dejar de confiar en la solucin de seguridad.

  • En base a las respuestas anteriores, Qu mtrica elegira para comparar modelos similares de clasificacin de phishing? Dado que los falsos negativos son ms peligrosos que los falsos positivos:

    • Recall (Sensibilidad o Tasa de Verdaderos Positivos):
      • Mide qu porcentaje de los correos o sitios de phishing fueron correctamente detectados.
      • Un recall alto significa que la mayora de los ataques de phishing sern identificados.
      • Sin embargo, un recall alto a veces se logra a costa de ms falsos positivos.
    • Precision (Precisin o Valor Predictivo Positivo):
      • Indica cuntos de los sitios que el modelo identific como phishing realmente lo eran.
      • Un precision alto reduce las falsas alarmas, evitando que el equipo de seguridad pierda tiempo en revisiones innecesarias.
    • AUC-ROC (rea Bajo la Curva ROC):
      • Evala la capacidad general del modelo para distinguir entre phishing y sitios legtimos.
      • Un AUC cercano a 1.0 indica un modelo excelente.

  • Qu modelo funcion mejor para la clasificacin de phishing? Por qu? El modelo Support Vector Machine:

    • Precisin general del 78.22%
    • Recall del 86% para phishing (detect la mayora de los sitios maliciosos)
    • Recall del 70% para sitios legtimos (algunos sitios normales fueron marcados como phishing)

    El SVM funcion bien en la deteccin de phishing, pero an tiene un problema con falsos positivos. Con la finalidad deminimizar los falsos negativosy capturar la mayor cantidad de phishing posible, SVM es una buena opcin. Sin embargo, podra mejorarse ajustando hiperparmetros.

    • TP (detectados correctamente) = 86% de 7,500 6,450 phishing detectados
    • FN (pasaron desapercibidos) = 7,500 - 6,450 1,050 phishing no detectados

Clculo de False Positives (FP) y True Negatives (TN)

  • Recall para sitios legtimos (70%) Indica que el modelo detecta correctamente el 70% de los correos legtimos.
    • TN (legtimos detectados correctamente) = 70% de 42,500 29,750 emails legtimos bien clasificados
    • FP (falsas alarmas, es decir, emails legtimos marcados como phishing) = 30% de 42,500 12,750 falsas alarmas

Funciona el modelo para este base rate del 15%? No del todo,

  • Genera demasiadas falsas alarmas (12,750 emails legtimos marcados como phishing), lo que sobrecarga al equipo de seguridad.
  • An deja pasar phishing (1,050 emails maliciosos no detectados), lo que representa un riesgo.

Cmo reducir la cantidad de falsas alarmas?

  • Aumentar la especificidad

    • Ajustar el umbral del modelo para reducir falsos positivos.
    • Usar tcnicas como calibracin de probabilidades para mejorar la confiabilidad del modelo.

  • Aplicar un segundo filtro

    • Utilizar otro modelo con alta precisin antes de marcar un email como phishing.
    • Implementar reglas heursticas adicionales (por ejemplo, verificar si la URL contiene dominios sospechosos o si el correo proviene de un remitente confiable).

  • Uso de listas blancas

    • Mantener una base de datos de emails y dominios confiables para reducir las falsas alarmas.
